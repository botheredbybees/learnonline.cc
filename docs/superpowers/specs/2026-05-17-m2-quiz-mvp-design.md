## Goal

A student can log in, pick a unit, work through its elements as quizzes, earn XP, and see a completion badge.

## Question Types

All three types are in scope for M2:

| Type | Grading | UI |
|---|---|---|
| Multiple choice | Auto (correct option index) | Click one of 4 options |
| Short answer | Auto (keyword matching against teacher-approved keywords) | Free text input |
| Ordering | Auto (correct sequence comparison) | Click steps in order (click-to-rank: first click = step 1, etc.) |

## Question Generation Pipeline

### Source material

Questions are generated by the **project author** (not by teacher instances) using a local Ollama instance. Source material:

- `lol_units.plain_english_description` â€” already generated for 47,733 of 47,788 AQTF units; covers all major training packages
- `unit_critical_aspects` (Knowledge Evidence items) â€” the AQTF content that drives question topics
- `unit_performance_criteria` â€” used to link questions to specific competency criteria

### Generation script

`backend/scripts/generate_questions.py --package MSF --model hermes3`

Per unit:
1. Pull `plain_english_description` from `lol_units` by code match
2. Pull `unit_critical_aspects` from learnonline DB as question source
3. For each KE item: call Ollama once to generate all applicable question types (MCQ + short answer always; ordering only for procedural PCs)
4. Write output to `questions-{PACKAGE}.json`

### Question pack format

References everything by AQTF code (never DB ID â€” IDs differ between installations):

```json
{
  "training_package": "MSF",
  "training_package_title": "Furnishing",
  "generated_at": "2026-05-17",
  "model": "hermes3",
  "qualifications": [...],
  "units": [{
    "code": "MSFFI2001",
    "title": "Install floor coverings",
    "description": "...",
    "plain_english_description": "...",
    "elements": [{
      "code": "01",
      "title": "Prepare for installation",
      "performance_criteria": [{"code": "01.01", "text": "..."}],
      "critical_aspects": [{"text": "Knowledge Evidence item..."}],
      "questions": [{
        "pc_code": "01.01",
        "question_type": "mcq",
        "question_text": "What PPE is required when handling solvent-based adhesives?",
        "options": {
          "choices": ["Safety glasses only", "Gloves and glasses", "Gloves, glasses and respirator", "No PPE required indoors"],
          "correct": 2,
          "explanation": "Solvent fumes require respiratory protection in addition to skin and eye protection."
        },
        "source": "ai_generated",
        "review_status": "draft"
      }]
    }]
  }]
}
```

Options payload by type:
- **MCQ:** `{"choices": [...], "correct": <index>, "explanation": "..."}`
- **Short answer:** `{"keywords": [...], "model_answer": "..."}`
- **Ordering:** `{"items": [...], "correct_order": [2, 0, 1]}`

### Distribution

One GitHub repository per training package under the `learnonline-cc` org (e.g., `github.com/learnonline-cc/questions-MSF`). Teachers download packs via the admin UI. The pack is self-contained â€” no TGA sync required for covered packages.

All questions imported from a pack land with `review_status = 'draft'` and `is_active = false`. They are invisible to students until a teacher approves them (individually or via bulk-approve in the Question Review tab).

## Backend API

All new routers follow existing patterns (FastAPI router, `get_db` from `database`, JWT via `JWTBearer`).

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/units/{id}/quiz-state` | Element list with pass/fail status for current user |
| GET | `/api/elements/{id}/questions` | Active approved questions for an element (shuffled) |
| POST | `/api/quiz/answer` | Submit one answer; returns `is_correct`, explanation, XP if element just passed |
| GET | `/api/units/{id}/progress` | User's element-by-element progress for a unit |
| POST | `/api/admin/packs/import` | Import a pack JSON: upserts unit structure + content + questions |
| GET | `/api/admin/questions` | List questions filtered by `review_status`, package, unit |
| PATCH | `/api/admin/questions/{id}` | Edit question text/options, approve/reject |
| GET | `/api/admin/progress` | Class progress â€” students Ă— elements for a unit |

### `POST /api/quiz/answer` scoring logic

1. Record `UserAnswer`
2. Check correctness (type-specific: index match / keyword match / sequence match)
3. If this completes all questions in the element correctly â†’ mark `user_element_progress` as passed, award XP
4. If all elements for the unit are now passed â†’ check for unit badge, award it
5. Return: `{is_correct, explanation, element_passed, xp_awarded, unit_completed, badge_awarded}`

## Student Quiz Flow

### Pages

| Page | Status | Purpose |
|---|---|---|
| `unit.html` | New | Unit detail â€” plain English description, element list with status, Start Quiz button |
| `quiz.html` | New | The quiz flow â€” one question at a time |
| `dashboard.html` | Extend | Add unit progress cards (X of Y elements passed per unit) |

### `unit.html` flow

1. Show unit `plain_english_description` (rich markdown rendered)
2. List elements with status indicator (not started / in progress / passed / failed)
3. "Start" / "Continue" / "Retry" button per element

### `quiz.html` flow

1. Load questions for the selected element (shuffled order)
2. For each question (one at a time):
   - Show element header: unit code Â· element title Â· XP on offer
   - Show progress: amber gradient bar ("2 of 5 questions")
   - Show question and answer UI (MCQ / short answer / click-to-rank ordering)
   - Student submits â†’ instant feedback (correct/incorrect + explanation)
   - Next question auto-advances after 1.5s on correct, stays for review on incorrect
3. After last question:
   - All correct â†’ element passed screen: XP awarded, amber celebration, "Next Element" button
   - Any wrong â†’ element failed screen: which questions failed, "Retry Element" button
4. After all elements passed â†’ unit complete screen: badge displayed, dashboard link

### Class Progress tab

## Gamification

- **XP per element:** 50 XP default (stored in `assessments.experience_points`, configurable per element). Awarded immediately when element passes via `POST /api/quiz/answer`.
- **Unit completion badge:** On all-elements-passed, backend looks for a badge with `code = unit_code`. If found, awards it. If not, awards a generic "Unit Complete" badge. Teacher creates unit-specific badges via existing badge admin.
- **Dashboard cards:** `dashboard.html` gains a "My Units" section â€” one card per unit the student has started, showing element progress bar and XP earned.
